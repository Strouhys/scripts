# Projekt: Retence dat na STG v BigQuery

## 1. Současné řešení v Teradatě

Retence dat je v současnosti řízena centrálně pomocí tabulky:

`ep_opr.MAN_TABLE_RETENTION`

V této tabulce může být pro libovolný objekt zapojený do datových loadů definováno retenční pravidlo. Pravidlo určuje, jakým způsobem se mají z dané tabulky odstraňovat stará data.

Typickým příkladem je mazání dat podle časového razítka, například:

* ponechat pouze data za posledních 30 dní,
* starší data automaticky odstranit.

Retenční pravidla jsou napojena přímo na:

* STG loady,
* údržbové uzly datových martů,
* EDW target loady.

Po dokončení běžného loadu se ověří, zda má cílová tabulka definované retenční pravidlo. Pokud ano, pravidlo se následně aplikuje.

## 2. Současný způsob spouštění retence

U STG loadů je retence řešena tak, že ji provede první batch, která se pro daný objekt v příslušném dni spustí.

Princip je následující:

1. Spustí se load dat.
2. Po dokončení loadu se zjistí, zda je pro tabulku definované retenční pravidlo.
3. Ověří se, zda již v daném dni retence proběhla.
4. Pokud ještě neproběhla, retenční pravidlo se vykoná.

Díky tomu by se retence neměla pro stejný objekt spouštět opakovaně během jednoho dne.

Existují také objekty, jejichž loady neběží každý den. Přesto je vhodné provádět retenci i nad těmito objekty, aby stará data nezůstávala v tabulkách jen proto, že se jejich standardní load delší dobu nespustil.

## 3. Problém v nové BigQuery platformě

Nová implementace datové platformy v BigQuery zatím retenci nijak systémově neřeší.

Je proto potřeba navrhnout nový způsob:

* evidence retenčních pravidel,
* plánování a spouštění retence,
* výběru objektů relevantních pro konkrétní běh,
* kontroly, zda a kdy byla retence provedena,
* monitoringu výsledků a chyb.

## 4. Předběžný směr řešení

Retenční maintenance joby by neměly být přímo spojeny s běžným zpracováním dat.

Místo současného modelu, kdy se retence spouští jako součást každého loadu, by mělo vzniknout samostatné maintenance řešení běžící nezávisle na standardních loadech.

Výhody tohoto přístupu:

* retence nebude zpomalovat nebo komplikovat běžné loady,
* bude možné zpracovat i tabulky, jejichž load neběží každý den,
* pravidla a jejich běhy budou řízeny centrálně,
* řešení půjde lépe monitorovat,
* jednotlivé retenční úlohy bude možné řídit paralelně.

## 5. Metadata a umístění konfigurace

Metadata potřebná pro řízení retence by měla být dostupná v BigQuery datasetu:

`opr_data`

Předpokládané řešení je následující:

* Cloud SQL bude obsahovat provozní metadata, například `data_stat`,
* nad těmito daty vznikne v `opr_data` view využívající federovaný dotaz do Cloud SQL,
* vedle tohoto view bude v `opr_data` samostatná tabulka s definicemi retenčních pravidel.

V `opr_data` by tedy mohly být dostupné například:

* informace o posledních loadech objektů,
* informace o předchozím provedení retence,
* definice retenčních pravidel,
* případně historie a výsledek jednotlivých retenčních běhů.

## 6. Navrhovaný způsob zpracování

Řešení by mělo dynamicky načíst všechny objekty, které mají definované retenční pravidlo.

Pro každý běh se následně:

1. načtou všechna aktivní retenční pravidla,
2. vyhodnotí se, které objekty jsou pro aktuální běh relevantní,
3. vyřadí se objekty, nad kterými se retence nyní provádět nemá,
4. z relevantních objektů se vytvoří jednotlivé retenční tasky,
5. tasky se následně spustí,
6. uloží se výsledek zpracování.

Důvody pro vyřazení objektu z konkrétního běhu mohou být například:

* retence již dnes proběhla,
* pravidlo není aktivní,
* tabulka neexistuje,
* objekt ještě nesplnil podmínku pro spuštění,
* předchozí retenční běh stále probíhá,
* objekt je dočasně blokovaný.

Je potřeba rozhodnout, zda každý objekt bude představovat samostatný task, nebo zda se objekty budou seskupovat do menšího počtu tasků.

## 7. Navrhovaná technologie

Jako první varianta se nabízí Python aplikace nebo skript.

Python by:

* načetl metadata a retenční pravidla,
* vybral objekty určené pro daný běh,
* vygeneroval příslušné BigQuery příkazy,
* vytvořil nebo spustil retenční tasky,
* zaznamenal stav a výsledek zpracování.

Python by byl v první fázi spouštěný pomocí systému `oflow`, pravděpodobně jednou denně.

Do budoucna by se stejné řešení mohlo přesunout do Airflow, aniž by bylo nutné zásadně měnit vlastní retenční logiku.

Předběžná architektura tedy je:

`oflow → Python orchestrátor → metadata v opr_data → retenční tasky v BigQuery`

Do budoucna:

`Airflow → Python / DAG → metadata v opr_data → retenční tasky v BigQuery`

## 8. Na čem jsme se předběžně shodli

Předběžně platí následující body:

* BigQuery retence bude řešena jako samostatná maintenance úloha.
* Retence nebude přímo napojena na běžné STG loady.
* Centrální metadata budou dostupná v datasetu `opr_data`.
* Provozní informace z Cloud SQL budou do BigQuery zpřístupněny pomocí federovaného view.
* Retenční pravidla budou pravděpodobně uložena v samostatné tabulce v `opr_data`.
* Řešení bude dynamicky vybírat pouze objekty relevantní pro konkrétní běh.
* Z vybraných objektů se budou generovat jednotlivé retenční tasky.
* První implementace může být provedena v Pythonu.
* Python se bude v první fázi spouštět pomocí `oflow`.
* Předpokládaná frekvence je jednou denně.
* Návrh má být připraven tak, aby jej bylo možné později přesunout do Airflow.

## 9. Otevřené otázky

Ještě je potřeba upřesnit především:

* přesnou strukturu tabulky retenčních pravidel,
* způsob zápisu samotné retenční podmínky,
* zda bude v pravidle uložen celý SQL příkaz, nebo pouze parametry,
* jak se bude evidovat poslední úspěšné provedení retence,
* zda se retence smí spustit vícekrát za den,
* jak se budou řešit chyby a opakované pokusy,
* jaký bude maximální počet souběžně spuštěných tasků,
* zda bude jeden task odpovídat jedné tabulce,
* jak se budou řešit partitionované a nepartitionované tabulky,
* zda bude podporované pouze mazání řádků, nebo také mazání partition,
* jak budou řešeny tabulky, které již neexistují,
* kdo a jak bude retenční pravidla zakládat nebo upravovat,
* jaké informace se budou ukládat do auditní historie,
* jak bude řešeno zabránění souběžného spuštění dvou maintenance běhů.
